# 百度网盘边下边播说明

最后更新：2026-04-02

这份文档只说明当前百度网盘“边下边播”这条链路已经做了什么，给后续的人和 agent 快速接手用。

## 1. 目标

目标不是把百度返回的 `m3u8` 直接扔给播放器，而是：

- 先由客户端解析百度播放入口
- 拿到真正的 segment 队列
- 自己控制下载、缓冲、失败处理和回收
- 最后只给播放器一个单一连续输入

## 2. 当前实际主链

当前实际播放链路是：

1. 百度播放入口先走 `step1`
2. 拿到 `adToken / adTime / ltime`
3. 再请求真正的 `m3u8`
4. 解析出 segment URL 列表
5. 进入 `BaiduPlaybackPipeline`
6. pipeline 下载并整理 segment
7. 返回一个本地虚拟播放入口：
   - `kidstv://session/<cacheKey>`
8. `libmpv` 通过 `stream_cb` 读取这个单一入口

也就是说：

- 播放器没有直接看到百度 `m3u8`
- 播放器也没有直接看到百度原始 segment URL

## 3. 已经做了什么

### 3.1 百度播放入口解析

已经处理了百度播放入口前面的两步流程：

- `streaming + nom3u8=1`
- `adToken` 重试
- `errno=133` 等待
- 拿到真正的 `m3u8`

并且不是把 `m3u8` 原样交出去，而是先解析成：

- `ResolvedBaiduSegment(url, duration)`

### 3.2 启动缓冲

当前做了首播缓冲：

- 启动门槛是前缀 `2` 个 segment 且累计至少 `4s`
- 启动前会优先并发 prime 前几个 segment
- 当前 prime 并发数是 `3`

目的就是：

- 不等整集下载完成
- 前面几段够播就先起

### 3.3 滚动预取

当前不是整集一次性下载，而是滚动推进：

- 播放位置会不断反馈给 pipeline
- pipeline 按当前播放位置计算后续预取目标
- 当前前向窗口大致是：
  - 约 `40s`
  - 或约 `96MB`

也就是说：

- 当前实现已经不是“整集下完再播”
- 而是“按播放进度继续往后拉”

### 3.4 旧数据回收

当前已经做了旧 segment 回收：

- 默认保留播放头后方约 `12s`
- 同时受总保留字节预算约束
- 当前上限约 `160MB`

所以它不是无限堆积缓存。

### 3.5 segment 下载和容错

每个 segment 目前都有独立处理：

- 带请求头下载
- 最多重试 `8` 次
- 记录下载耗时和速度
- 判断返回内容是不是媒体数据

当前会识别几类 payload：

- `transport stream`
- `mp4/fmp4-like`
- `json`
- `text`
- `unknown binary`

如果百度回的是 JSON / 文本错误体，不会再误当成视频片段继续吃。

### 3.6 片段整理

当前这层已经不再假设百度下回来的东西一定是标准 `.ts`。

处理策略是：

- 如果 payload 本身就是 `TS`，直接作为播放片段使用
- 如果 payload 更像 `MP4/fMP4`，先落盘，再用 FFmpeg transmux 成 `.ts`

所以现在这层已经具备了：

- “百度给的是 mp4-like 片段，也能继续喂播放器”的能力

### 3.7 mpv 单一入口

播放器侧当前已经改成单一入口：

- 播放器只打开 `kidstv://session/<cacheKey>`
- `mpv stream_cb` 的 `open/read/seek/close` 都走这条会话流

当前行为是：

- `read` 没数据时会等待
- `read` 过程中会反向触发后续预取
- `seek` 只允许在当前已缓冲范围内进行
- `close` 时关闭对应 pipeline session

## 4. 现在这版“边下边播”到底意味着什么

可以这样理解当前实现：

- 百度负责给播放入口和片段 URL
- 我们自己负责把片段下载下来并整理成播放器更稳定的输入
- `mpv` 不直接消费百度原始播放地址

它已经不是：

- `ffconcat`
- 整集下载后再播
- 多播放器轮流试
- 把百度 m3u8 直接丢给播放器碰运气

## 5. 当前还没做到的事

这版虽然已经是“边下边播”，但还没有做到最强形态。

还没做到的重点有：

- 还不是容器级连续重组输出
- 当前更像“逐片整理后，通过单一虚拟流供给 mpv”
- seek 目前还是受已缓冲范围限制，不是完整随机访问
- 首播等待仍然会被百度入口等待和前几个 segment 的 transmux 成本影响
- 百度 token 目前没有做完整自动 refresh 闭环

## 6. 当前最明显还没解决的问题：有些视频下载很慢

这个问题当前没有彻底解决。

现象是：

- 有些视频起播很快
- 有些视频会卡在首播前等待很久
- 单个 segment 的下载速度会明显波动
- 有时会掉到很低的速度

目前确认到的慢点主要有两层：

### 6.1 播放入口本身慢

在真正开始下载 segment 之前，百度播放入口本身就可能很慢：

- 先要走 `step1`
- 可能要等待 `adTime / ltime`
- 可能多次遇到 `errno=133`
- 有时要多轮请求之后才能拿到真正的 `m3u8`

这部分慢，不属于播放器，也不属于 segment 下载本身。

### 6.2 segment 下载本身慢

即使已经拿到了 `m3u8`，后面的 segment 也不一定稳定：

- 某些 segment 很快
- 某些 segment 明显很慢
- 同一条视频内部就会波动
- 不同视频之间波动也很大

这会直接影响：

- 首播等待
- 播放过程中的连续性
- 缓冲体验

### 6.3 当前为什么还会感觉慢

虽然现在已经不是“整集下载后再播”，但首播还是会被这些因素拖慢：

- 百度入口等待
- 前几个 segment 的下载速度
- 前几个 segment 的整理/transmux 成本
- 当前 pipeline 还没有把“慢 segment 规避”和“更激进首播调度”做到足够强

所以当前状态更准确的说法是：

- 主链已经对了
- 但“商业级首播速度和稳定性”还没打磨到位

## 7. 面向商业级产品，目标应该是什么

如果目标是接近网易爆米花这类产品，不能把当前版本当终点。

更合理的产品目标应该是：

- 一个播放器内核
- 一层真正强的媒体消费层 / pipeline
- 所有网盘脏活都留在 pipeline
- 播放器只吃统一的连续输入

### 7.1 对爆米花的判断

这里是工程判断，不是拿到了对方源码。

基于它公开宣称的能力，比较合理的推断是：

- 它不太像“把百度 m3u8 直接扔给系统播放器”
- 更像是“强播放器内核 + 自研媒体供给层”
- 不同来源在播放器前面应该都有一层适配和整理

这类产品通常要同时解决：

- 多来源接入
- 非标准播放入口
- 首播速度
- 长时间稳定播放
- 拖动恢复
- 字幕、音轨、HDR、杜比等能力

### 7.2 对我们当前项目意味着什么

当前这条百度链路只是第一步，意义在于：

- 方向终于收敛到了对的主线上
- 不再靠弱播放器或临时拼接方案碰运气

但如果目标真的是商业级产品，后面要继续把这条线做厚：

- 更强的首播调度
- 更强的 segment 并发与超时重试
- 更稳的滚动缓冲
- 更完整的 seek 能力
- 更强的媒体重组 / 容器处理
- 更完整的 token / 错误恢复闭环

简短说：

- 现在已经不是玩具路线
- 但离“爆米花级别体验”还有明显距离
- 正确做法不是换回弱播放器，而是继续把 pipeline 和单一播放器内核做强

## 8. 给后续接手的人一句话

当前这条百度链路，最重要的结论是：

- 方向已经切到对的主线上了：
  - `百度入口解析 + 自建 pipeline + libmpv 单一入口`

后续继续做时，应该沿着这条线加厚：

- 更强的缓冲调度
- 更快的首播
- 更稳的 seek
- 更强的片段兼容
- 以及更完整的媒体重组能力

不要再回到：

- 直接把百度 m3u8 丢给播放器
- `ffconcat`
- 整集下完再播
- 多播放器混试
